## 各層の評価詳細

### サンドボックス（OS レベル）: 未導入 → 案内済み

**現状:**
- `/sandbox` コマンド / `settings.json` の `"sandbox": {"enabled": true}` は未使用
- Docker隔離も未使用
- セットアップドキュメント v3 に導入手順を記載済み

**サンドボックスでしか守れないもの:**
- 未知の攻撃パターン（フックのパターンマッチに依存しない）
- ネットワーク通信の完全遮断（curl/wgetでの外部送信）
- プロジェクト外ファイルへのアクセス制限

**サンドボックスの限界:**
- macOS版は実験的機能で完全ではない
- 正当な外部通信（npm install, git push等）も制限される
- 開発体験とのトレードオフがある

**評価:** フック/ルールとは別レイヤーの防御。併用が理想だが、開発体験を考慮すると必須ではない。

## 残存する弱点一覧

### 優先度: 高

#### 1. settings.local.json の自動allow蓄積問題

- **現状**: 修正済みだが、会話中に再度追加される可能性あり
- **対策案**: 月1回の定期清掃を習慣化。確認コマンドをドキュメントに記載済み
- **ステータス**: 運用で対応（自動化は困難）

#### 2. フックの分割書き込みバイパス

- **現状**: 未対策
- **リスク**: Claudeが2回に分けてAPIキーを書くとすり抜ける
- **対策案**: PostToolUse フックでファイル全体スキャン（動作速度とのトレードオフ）
- **ステータス**: 要検討

### 優先度: 中

#### 3. サンドボックス未導入

- **現状**: 案内済みだが実際には未使用
- **リスク**: ネットワーク制限なし、プロジェクト外アクセス可能
- **対策案**: 機密性の高い作業時のみ `/sandbox` コマンドで有効化
- **ステータス**: ユーザー判断

#### 4. SessionStart再注入の内容

- **現状**: 1行要約のみ
- **対策案**: 主要ルール3-5行に拡充
- **ステータス**: 低優先

### 優先度: 低（運用面）

#### 5. インシデント対応手順の未策定

- **対策案**:
  - 漏洩発見時: 該当キーを即座に無効化
  - GitHub: Settings → Developer settings → revoke
  - Google Cloud: Console → 認証情報 → 削除
  - freee: アプリ設定 → トークン再発行
  - git履歴: `git filter-branch` または BFG Repo-Cleaner
- **ステータス**: 未着手

#### 6. キーローテーション指針

- **対策案**: 90日ごとのローテーション推奨
- **ステータス**: 未着手
